# PRD (Product Requirements Document)

## Engineering Capability Growth Framework (ECGF)

---

## 1. 제품 개요 (Overview)

### 1.1 제품명
**Engineering Capability Growth Framework (ECGF)**

### 1.2 배경
- 엔지니어 연차 분포가 1년차 ~ 30년차 이상으로 매우 넓음
- 동일한 교육/기대 수준은 비효율적
- 관리자 개인 역량에 의존한 육성 방식의 한계
- 기술 전문가와 리더 트랙이 명확히 구분되지 않음

### 1.3 목표
- 연차에 맞는 명확한 역량 기준 정의
- 엔지니어 성향별 전문가 성장 경로 제공
- 개인의 성장 = 조직의 기술 자산 축적

---

## 2. 문제 정의 (Problem Statement)

| 문제 | 설명 |
|------|------|
| 성장 기준 불명확 | "잘한다"의 정의가 사람마다 다름 |
| 연차별 역할 혼재 | 시니어가 실무만, 주니어가 방향 고민 |
| 기술 단절 | 경험과 노하우가 구조적으로 전수되지 않음 |
| 동기 저하 | 미래 커리어가 보이지 않음 |

---

## 3. 사용자 정의 (User Personas)

### 3.1 주니어 엔지니어 (1~3년)
- **특징**: 학습 의욕 높음, 방향성 부족
- **니즈**: 기본기, 빠른 피드백, 성장 로드맵

### 3.2 미들 엔지니어 (4~10년)
- **특징**: 실무 중심, 전문성 갈림길
- **니즈**: 기술 깊이, 선택 가능한 트랙

### 3.3 시니어 엔지니어 (11~20년)
- **특징**: 문제 해결 능력 우수, 영향력 확대 필요
- **니즈**: 기술 리더십, 후배 육성 역할

### 3.4 마스터/수석 (20년+)
- **특징**: 조직의 기술 자산
- **니즈**: 전략, 아키텍처, 외부 영향력

---

## 4. 엔지니어 성향 분류 (Engineering Archetypes)

| 성향 | 설명 |
|------|------|
| 문제 해결형 | 장애, 트러블슈팅, Root Cause 분석 |
| 설계/아키텍처형 | 구조 설계, 확장성, 표준화 |
| 연구/개선형 | 성능, 품질, 자동화, 신기술 |
| 현장/운영형 | 안정성, 운영 최적화, 대응 |
| 리더/멘토형 | 기술 전파, 리뷰, 의사결정 |

> **참고**: 복수 성향 허용 (Primary / Secondary)

## 5. 연차별 역량 기대치 (Career Levels)

### L1: Foundation (1~3년)
- **기술**: 기본 원리 이해, 표준 준수
- **업무**: 지시 기반 실행
- **기대 산출물**: 문서화, 재현 가능한 작업

### L2: Practitioner (4~7년)
- **기술**: 독립적 문제 해결
- **업무**: 기능 단위 책임
- **기대 산출물**: 개선 제안, 코드/설계 리뷰 참여

### L3: Senior Engineer (8~12년)
- **기술**: 복합 문제 해결, 설계 주도
- **업무**: 프로젝트 기술 책임
- **기대 산출물**: 기술 가이드, 표준 정립

### L4: Staff/Principal (13~20년)
- **기술**: 시스템 관점, 장기적 의사결정
- **업무**: 기술 방향성 제시
- **기대 산출물**: 아키텍처, 기술 전략

### L5: Master (20년+)
- **기술**: 조직 외부까지 영향
- **업무**: 기술 문화·철학 수립
- **기대 산출물**: 기술 백서, 외부 발표

---

## 6. 성장 메커니즘 (Growth Mechanism)

1. **연차별 역량 매트릭스**
2. **분기별 성장 목표 설정** (OKR 형태)
3. **기술 리뷰 & 회고**
4. **멘토-멘티 제도** (역멘토링 포함)
5. **기술 자산 문서화** (Wiki, ADR)

---

## 7. 성공 지표 (Success Metrics)

- 연차별 역량 달성률
- 기술 문서/자산 증가량
- 내부 기술 만족도
- 핵심 인력 이탈률 감소
- 기술 의사결정 리드 타임 감소

---

## 8. 연차별 역량 매트릭스 (체크리스트)

### 공통 역량 축
- 기술 역량
- 문제 해결
- 협업/커뮤니케이션
- 영향력/리더십
- 기술 자산화

### L1 – Foundation (1~3년)

#### 기술
- [ ] 기본 원리 이해 (왜 이렇게 동작하는지 설명 가능)
- [ ] 사내 표준/가이드 숙지
- [ ] 기존 코드/시스템 수정 가능

#### 문제 해결
- [ ] 문제 재현 가능
- [ ] 원인 가설 제시 가능
- [ ] 상위자 도움을 받아 해결

#### 협업
- [ ] 진행 상황 공유
- [ ] 질문을 정리해서 요청

#### 기술 자산화
- [ ] 작업 내용 문서화
- [ ] 장애/이슈 정리 가능

---

### L2 – Practitioner (4~7년)

#### 기술
- [ ] 독립적으로 기능 구현
- [ ] 기술 선택의 장단점 설명
- [ ] 성능/품질 고려한 설계

#### 문제 해결
- [ ] Root Cause 분석 가능
- [ ] 재발 방지책 제안

#### 협업
- [ ] 코드/설계 리뷰 참여
- [ ] 타 부서 협업 경험

#### 기술 자산화
- [ ] 개선 사례 문서화
- [ ] 사내 기술 공유 발표

---

### L3 – Senior Engineer (8~12년)

#### 기술
- [ ] 시스템 단위 설계 주도
- [ ] 복잡한 장애 해결 리드

#### 문제 해결
- [ ] 구조적 문제 정의
- [ ] 장기적 개선 로드맵 제시

#### 협업
- [ ] 프로젝트 기술 책임
- [ ] 주니어 멘토링

#### 기술 자산화
- [ ] 기술 가이드 작성
- [ ] 표준/베스트 프랙티스 정립

---

### L4 – Staff / Principal (13~20년)

#### 기술
- [ ] 아키텍처 설계
- [ ] 기술 방향성 결정

#### 문제 해결
- [ ] 조직 차원의 문제 해결
- [ ] 리스크 사전 제거

#### 협업
- [ ] 기술 의사결정 조율
- [ ] 복수 팀 영향력

#### 기술 자산화
- [ ] 핵심 기술 문서화
- [ ] 사내 교육 설계

---

### L5 – Master (20년+)

#### 기술
- [ ] 기술 철학/비전 제시
- [ ] 외부 기술 트렌드 선도

#### 영향력
- [ ] 조직 문화 형성
- [ ] 외부 발표/기고

## 9. 1:1 면담 질문 가이드

### 공통 구조 (분기 1회 권장)
1. 지난 분기 돌아보기
2. 기술 성장
3. 어려움/장애
4. 다음 분기 목표
5. 관리자 지원 요청

### L1 질문 예시
- 가장 이해가 안 갔던 기술은?
- 최근 실수에서 배운 점은?
- 혼자 해결해보고 싶은 영역은?

### L2 질문 예시
- 요즘 가장 자신 있는 기술은?
- 기술 선택에서 고민했던 지점은?
- 다음 단계(Senior)로 가기 위해 부족한 점은?

### L3 질문 예시
- 기술적으로 가장 큰 결정은?
- 후배에게 어떤 도움을 주고 있는가?
- 우리 시스템의 가장 큰 구조적 문제는?

### L4~L5 질문 예시
- 조직 기술의 가장 큰 리스크는?
- 앞으로 2~3년 기술 방향은?
- 후계자(차세대 리더)는 누구라고 생각하는가?

## 10. 전문가 트랙 vs 관리 트랙 설계

### 공통 원칙
- 보상/승진 동일
- 상호 이동 가능
- 강제 분리 금지 (선택)

### 🔧 전문가 트랙 (Individual Contributor)

| 레벨 | 역할 |
|------|------|
| Senior | 기술 문제 해결 리드 |
| Staff | 시스템/아키텍처 책임 |
| Principal | 조직 기술 전략 |
| Master | 기술 철학/대외 영향 |

**평가 기준**
- 기술 난이도
- 문제 해결 임팩트
- 기술 자산화 기여도

### 🧭 관리 트랙 (People Manager)

| 레벨 | 역할 |
|------|------|
| Tech Lead | 기술 + 사람 |
| Engineering Manager | 팀 성과 |
| Head / Director | 조직 전략 |

**평가 기준**
- 팀 성과
- 인재 성장
- 이탈률
- 협업 품질

### 선택 가이드 (면담 시 활용)
- "사람 문제 vs 기술 문제 중 뭐가 더 에너지가 나는가?"
- "성과를 개인 vs 팀 중 어디서 느끼는가?"

## 11. 평가 · 승진 기준 연계

### 평가 구조 (예시)

| 항목 | 비중 |
|------|------|
| 기술 역량 | 40% |
| 문제 해결 | 25% |
| 협업/리더십 | 20% |
| 기술 자산화 | 15% |

### 승진 판단 기준
- ❌ 연차 충족 (필수 아님)
- ✅ 역량 매트릭스 80% 이상 충족
- ✅ 상위 레벨 역할을 이미 수행 중

### 관리자 체크 질문
1. 이미 한 단계 위 역할을 하고 있는가?
2. 없어지면 조직이 흔들리는 기술인가?
3. 이 사람이 키운 사람이 있는가?

---

## 12. 마무리

이 체계의 핵심은  
👉 **"사람을 평가하기 위한 문서가 아니라, 사람을 키우기 위한 구조"** 입니다.
