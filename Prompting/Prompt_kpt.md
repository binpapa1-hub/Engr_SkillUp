## 📋 회고 개요

### 목적
프로젝트 진행 과정에서의 성과와 개선점을 정리하고, 다음 단계를 위한 액션 아이템을 도출하기 위한 회고입니다.

### 회고 방법론
**KPT (Keep, Problem, Try)**
- **Keep**: 잘하고 있는 것, 유지해야 할 것
- **Problem**: 문제점, 개선이 필요한 것
- **Try**: 다음에 시도해볼 것

## ⚠️ Problem (문제점 및 개선 필요 사항)

### 1. 머지 후 동작 불가 문제

#### 문제 상황
- Green 브랜치를 main 브랜치에 머지한 후 화면은 표시되지만 기능이 동작하지 않음
- 여러 이슈가 동시에 발생하여 디버깅 시간 소요

#### 발견된 문제점
1. **Import 경로 불일치**
   - `app.js`에서 `validateMember`를 `validation.js`에서 import 시도
   - 실제로는 `member.js`에 정의되어 있음

2. **함수명 불일치**
   - `app.js`에서 `deleteMember`를 import 시도
   - 실제로는 `removeMember`로 정의됨

3. **데이터 타입 변환 누락**
   - 폼에서 입력받은 `years` 값이 문자열로 처리됨
   - `parseInt` 변환 누락으로 검증 로직 실패

4. **중복 Import**
   - `index.html`에서 `handleTabChange`를 두 번 import

5. **탭 전환 로직 문제**
   - `showTab` 함수가 클릭한 탭 버튼을 올바르게 활성화하지 못함

#### 해결 과정
- 리팩토링 단계에서 모든 문제 해결
- 수정된 파일: `src/app.js`, `src/ui.js`, `index.html`

#### 개선 방안
- **코드 리뷰 강화**: 머지 전 코드 리뷰 필수화
- **통합 테스트 자동화**: 머지 후 자동 테스트 실행
- **타입 안정성**: TypeScript 도입 검토

### 2. 타입 안정성 부족

#### 문제 상황
- JavaScript의 동적 타입 특성으로 인한 런타임 에러 가능성
- 데이터 타입 변환 누락 (예: `years` 문자열 처리)

#### 개선 방안
- **TypeScript 도입**: 점진적 마이그레이션 검토
- **JSDoc 타입 주석**: 최소한 타입 주석 보강
- **런타임 타입 검증**: `validateMember` 함수 강화

### 3. 에러 처리 일관성 부족

#### 문제 상황
- 일부 함수는 예외를 throw하고, 일부는 boolean 반환
- 사용자 친화적인 에러 메시지 부족

#### 개선 방안
- **에러 처리 표준화**: 통일된 에러 처리 패턴 정의
- **에러 메시지 개선**: 사용자 친화적인 메시지 작성
- **에러 로깅 시스템**: 에러 추적 및 분석 도구 도입

### 4. CI/CD 파이프라인 미구축

#### 문제 상황
- 수동으로 테스트 실행
- 머지 후 자동 검증 부재

#### 개선 방안
- **GitHub Actions 설정**: 자동 테스트 실행
- **자동 배포 파이프라인**: 배포 자동화
- **코드 품질 검사**: ESLint, Prettier 자동 실행

### 5. 사용자 테스트 부재

#### 문제 상황
- 개발자 관점의 테스트만 존재
- 실제 사용자 시나리오 검증 부족

#### 개선 방안
- **사용자 테스트 시나리오**: 실제 사용 케이스 테스트
- **UX 테스트**: 사용자 경험 검증
- **피드백 수집**: 실제 사용자 피드백 반영
