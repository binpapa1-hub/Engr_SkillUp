# Engineering Capability Growth Framework (ECGF)

## 📌 목적
엔지니어 기술부서 구성원을 연차와 성향에 맞게 체계적으로 성장시켜  
개인의 전문성과 조직의 기술 경쟁력을 동시에 강화한다.

---

## 🧭 기본 원칙
- 연차 ≠ 역량 (하지만 기준은 필요하다)
- 모든 엔지니어는 전문가 트랙을 가진다
- 기술 자산은 개인이 아니라 조직의 것이다
- 성장 책임은 개인 + 조직 공동 책임

---

## 🧑‍💻 엔지니어 성장 단계

| Level | 연차 | 역할 |
|----|----|----|
| L1 | 1~3년 | Foundation |
| L2 | 4~7년 | Practitioner |
| L3 | 8~12년 | Senior Engineer |
| L4 | 13~20년 | Staff / Principal |
| L5 | 20년+ | Master |

---

## 🔍 엔지니어 성향 트랙

- **문제 해결형**: 장애, 트러블슈팅, Root Cause 분석
- **설계/아키텍처형**: 구조 설계, 확장성, 표준화
- **연구/개선형**: 성능, 품질, 자동화, 신기술
- **운영/현장형**: 안정성, 운영 최적화, 대응
- **리더/멘토형**: 기술 전파, 리뷰, 의사결정

> 각 엔지니어는 Primary / Secondary 성향을 가진다.

---

## 🚀 성장 방법

### 1. 연차별 역량 기준
- 기술
- 문제 해결
- 협업
- 영향력

### 2. 분기별 성장 목표
- 개인 OKR 수립
- 관리자 1:1 리뷰

### 3. 기술 자산화
- 설계 문서
- 장애 보고서
- 기술 가이드

### 4. 멘토링
- 시니어 → 주니어
- 주니어 → 시니어 (신기술)

## 📋 연차별 역량 매트릭스

### L1 – Foundation (1~3년)
- 기본 원리 이해
- 문제 재현 및 가설 제시
- 작업 내용 문서화

### L2 – Practitioner (4~7년)
- 독립적 기능 구현
- Root Cause 분석
- 개선 사례 문서화

### L3 – Senior Engineer (8~12년)
- 시스템 단위 설계 주도
- 구조적 문제 정의
- 기술 가이드 작성

### L4 – Staff / Principal (13~20년)
- 아키텍처 설계
- 조직 차원 문제 해결
- 핵심 기술 문서화

### L5 – Master (20년+)
- 기술 철학/비전 제시
- 조직 문화 형성
- 외부 발표/기고

## 💬 1:1 면담 가이드

### 공통 구조
1. 지난 분기 돌아보기
2. 기술 성장
3. 어려움/장애
4. 다음 분기 목표
5. 관리자 지원 요청

### 레벨별 질문 예시
- **L1**: 가장 이해가 안 갔던 기술은?
- **L2**: 요즘 가장 자신 있는 기술은?
- **L3**: 기술적으로 가장 큰 결정은?
- **L4~L5**: 조직 기술의 가장 큰 리스크는?

## 🎯 전문가 트랙 vs 관리 트랙

### 🔧 전문가 트랙 (IC)
- Senior → Staff → Principal → Master
- 기술 난이도, 문제 해결 임팩트 평가

### 🧭 관리 트랙 (Manager)
- Tech Lead → Engineering Manager → Head/Director
- 팀 성과, 인재 성장 평가

> 보상/승진 동일, 상호 이동 가능

---

## 📊 평가 · 승진 기준

### 평가 비중
- 기술 역량: 40%
- 문제 해결: 25%
- 협업/리더십: 20%
- 기술 자산화: 15%

### 승진 판단
- ✅ 역량 매트릭스 80% 이상 충족
- ✅ 상위 레벨 역할을 이미 수행 중
- ❌ 연차는 필수 아님

## 🎯 기능/비기능 요구사항

### 기능 요구사항

#### 1. 부서원 관리 시스템
- **1.1 부서원 등록**
  - 이름, 주요 성향(Primary), 보조 성향(Secondary), 근무 연차, 업무 레벨 입력
  - 필수 항목 검증 (이름, 주요 성향, 연차, 레벨)
  - 데이터 유효성 검증 (연차 범위: 0~50년)
  
- **1.2 부서원 목록 조회**
  - 등록된 모든 부서원을 카드 형태로 표시
  - 레벨별 정렬 (L5 → L1), 연차 내림차순
  - 배지 형태로 레벨, 연차, 성향 표시
  
- **1.3 부서원 정보 수정**
  - 목록에서 수정 버튼 클릭 시 폼에 기존 정보 자동 입력
  - 모든 항목 수정 가능
  - 수정 후 저장 시 목록 자동 갱신
  
- **1.4 부서원 삭제**
  - 삭제 버튼 클릭 시 확인 대화상자 표시
  - 확인 시 삭제 완료 및 목록 갱신

#### 2. 정보 조회 시스템
- **2.1 개요 탭**: 프로젝트 목적, 기본 원칙, 성장 방법, 성공 지표, 최종 목표 표시
- **2.2 연차별 역량 탭**: L1~L5 각 레벨별 상세 역량 체크리스트 (기술, 문제 해결, 협업, 기술 자산화)
- **2.3 성향 트랙 탭**: 5가지 엔지니어 성향 소개 및 설명
- **2.4 면담 가이드 탭**: 공통 면담 구조 및 레벨별 질문 예시
- **2.5 트랙 비교 탭**: 전문가 트랙(IC) vs 관리 트랙(Manager) 비교
- **2.6 평가 기준 탭**: 평가 구조, 승진 판단 기준, 관리자 체크 질문

#### 3. 사용자 인터페이스
- **3.1 탭 네비게이션**: 7개 탭 간 전환 (개요, 부서원 관리, 연차별 역량, 성향 트랙, 면담 가이드, 트랙 비교, 평가 기준)
- **3.2 반응형 디자인**: 데스크톱, 태블릿, 모바일 대응
- **3.3 시각적 요소**: 카드 레이아웃, 배지, 그라데이션 배경

#### 4. 데이터 관리
- **4.1 로컬 스토리지**: 브라우저 로컬 스토리지에 부서원 데이터 저장
- **4.2 데이터 영속성**: 페이지 새로고침 후에도 데이터 유지

### 비기능 요구사항

#### 1. 성능
- 페이지 로드 시간: 2초 이내
- 탭 전환 애니메이션: 부드러운 전환 (0.5초)
- 데이터 저장/조회: 즉시 반영

#### 2. 사용성
- 직관적인 UI/UX
- 명확한 에러 메시지
- 삭제 시 확인 대화상자로 실수 방지

#### 3. 호환성
- 최신 브라우저 지원 (Chrome, Firefox, Safari, Edge)
- 모바일 브라우저 지원
- 로컬 스토리지 지원 필수

#### 4. 유지보수성
- 코드 구조화 및 모듈화
- 명확한 주석 및 문서화

## 📖 기능별 시나리오

### 시나리오 1: 부서원 등록

**전제 조건**: 사용자가 "부서원 관리" 탭에 접근

**시나리오 1.1: 정상 등록**
1. 사용자가 이름 입력: "홍길동"
2. 주요 성향 선택: "문제 해결형"
3. 보조 성향 선택: "설계/아키텍처형" (선택사항)
4. 근무 연차 입력: "5"
5. 업무 레벨 선택: "L2 - Practitioner"
6. "등록" 버튼 클릭
7. **예상 결과**: 
   - 폼이 초기화됨
   - 부서원 목록에 "홍길동" 카드가 추가됨
   - 로컬 스토리지에 데이터 저장됨

**시나리오 1.2: 필수 항목 누락**
1. 사용자가 이름만 입력하고 "등록" 버튼 클릭
2. **예상 결과**: 
   - 브라우저 기본 유효성 검사로 필수 항목 입력 요구
   - 등록되지 않음

**시나리오 1.3: 잘못된 연차 입력**
1. 사용자가 연차에 음수(-5) 입력
2. **예상 결과**: 
   - 브라우저 기본 유효성 검사로 범위 초과 경고
   - 등록되지 않음

### 시나리오 2: 부서원 목록 조회

**전제 조건**: 등록된 부서원이 3명 이상 존재

**시나리오 2.1: 목록 표시**
1. 사용자가 "부서원 관리" 탭 클릭
2. **예상 결과**: 
   - 등록된 모든 부서원이 카드 형태로 표시됨
   - 레벨 순서로 정렬됨 (L5 → L1)
   - 각 카드에 이름, 레벨 배지, 연차 배지, 성향 배지 표시됨

**시나리오 2.2: 빈 목록**
1. 등록된 부서원이 없는 상태
2. **예상 결과**: 
   - "등록된 부서원이 없습니다" 메시지 표시
   - 폼 사용 안내 메시지 표시

### 시나리오 3: 부서원 정보 수정

**전제 조건**: 등록된 부서원 "홍길동" 존재

**시나리오 3.1: 정보 수정**
1. "홍길동" 카드의 "수정" 버튼 클릭
2. **예상 결과**: 
   - 폼에 기존 정보가 자동으로 입력됨
   - 연차를 "5"에서 "6"으로 변경
   - "등록" 버튼 클릭
3. **최종 결과**: 
   - 목록의 "홍길동" 정보가 업데이트됨
   - 로컬 스토리지에 변경사항 저장됨

### 시나리오 4: 부서원 삭제

**전제 조건**: 등록된 부서원 "홍길동" 존재

**시나리오 4.1: 삭제 확인 후 삭제**
1. "홍길동" 카드의 "삭제" 버튼 클릭
2. **예상 결과**: 
   - 확인 대화상자 표시 ("정말 삭제하시겠습니까?")
   - "확인" 클릭
3. **최종 결과**: 
   - "홍길동" 카드가 목록에서 제거됨
   - 로컬 스토리지에서 데이터 삭제됨

**시나리오 4.2: 삭제 취소**
1. "홍길동" 카드의 "삭제" 버튼 클릭
2. 확인 대화상자에서 "취소" 클릭
3. **예상 결과**: 
   - 삭제되지 않음
   - 목록에 "홍길동" 카드가 그대로 유지됨

### 시나리오 5: 탭 네비게이션

**시나리오 5.1: 탭 전환**
1. 현재 "개요" 탭에 있음
2. "연차별 역량" 탭 클릭
3. **예상 결과**: 
   - "개요" 탭 비활성화
   - "연차별 역량" 탭 활성화
   - 해당 콘텐츠가 페이드인 애니메이션과 함께 표시됨

**시나리오 5.2: 부서원 관리 탭 접근**
1. 다른 탭에서 "부서원 관리" 탭 클릭
2. **예상 결과**: 
   - 부서원 목록이 자동으로 렌더링됨
   - 등록 폼과 목록이 표시됨

### 시나리오 6: 데이터 영속성

**시나리오 6.1: 페이지 새로고침**
1. 부서원 3명 등록
2. 페이지 새로고침 (F5)
3. **예상 결과**: 
   - 등록된 3명의 부서원이 목록에 그대로 표시됨
   - 로컬 스토리지 데이터가 유지됨

**시나리오 6.2: 브라우저 종료 후 재접속**
1. 부서원 등록 후 브라우저 종료
2. 브라우저 재실행 후 페이지 접속
3. **예상 결과**: 
   - 이전에 등록한 부서원 정보가 목록에 표시됨
